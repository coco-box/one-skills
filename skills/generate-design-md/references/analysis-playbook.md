# 设计语言分析手册

## 从页面提炼系统

先识别“辨识度杠杆”，再补齐 token。辨识度通常来自 2 至 4 个稳定关系，例如画布与主色的对比、标题字体与正文字体的搭配、卡片边界方式、内容密度、独特的装饰机制。没有这些关系，精确色值也只会生成通用模板。

依次分析：

1. 视觉主题：克制或外放、编辑感或工具感、稀疏或密集、平面或有层次。
2. 颜色角色：画布、表面、正文、弱化文本、边框、主操作、语义状态、深色表面和装饰色。
3. 字体系统：字体类别、字号层级、字重、行高、字距、数字和代码的特殊处理。
4. 空间与布局：基础单位、内容宽度、栏数、section 节奏、卡片内距和对齐原则。
5. 形状与层次：圆角阶梯、边框、阴影、模糊、叠层和深浅表面切换。
6. 组件语法：按钮、导航、卡片、表单、标签、表格、弹窗和反馈状态如何组合上述 token。
7. 响应式行为：何时折叠、重排、横向滚动、隐藏、缩小或保持固定尺寸。
8. 动效与素材：持续时间、缓动、进入方式、图片比例、裁切和图标风格。只有观察到稳定规律时才写入。

完成归纳后再做一次覆盖检查。`awesome-design-md` 的成熟样例通常不仅列出基础 token，还会把颜色按 Brand/Surface/Text/Semantic 分组，把字体写成 9 至 15 个有角色的层级，并分别说明容器、网格、触控目标、组件状态、签名组件、折叠策略和已知缺口。来源确有证据时应达到同类覆盖；来源缺失时写入 Known Gaps 或 `Source Scope & Confidence`，不要为了凑齐目录编造规则。

## 归纳规则

- 为功能角色命名，如 `canvas`、`ink-muted`、`primary-hover`，不要使用 `gray-3` 一类无语义名称，除非原项目就是分级色板。
- 合并视觉上和用途上相同的近似值；保留有明确功能差异的相邻值。
- 间距优先归纳为 4px 或 8px 基数的有限阶梯；局部例外写在组件说明中。
- 字体 token 至少包含 display、body、caption、button；代码型产品再加入 mono/code。
- 组件 token 只放稳定、可复用且能被格式表达的属性。复杂布局、边框、动效和素材规则放在 Markdown。
- 至少区分品牌/交互色、表面色、文本色和语义色；来源未观察到 success、warning、error 时明确缺口，不生成伪精确 token。
- 字体层级应覆盖实际出现的 display、headline/title、body、label/caption、button/nav，技术型产品按证据加入 mono/code。不要机械追求数量，但要让常见页面无需临时发明字号。
- 对核心组件同时检查默认、hover、focus、active/pressed、disabled、loading、error、empty；只记录已观察状态，缺失状态归入 Known Gaps。
- 将来源独有的导航、内容编排、签名组件和素材几何单列说明，避免只有通用按钮/卡片/输入框。
- Do/Don't 必须具体到可执行判断，例如“卡片依靠 1px hairline 分隔，不使用重阴影”，不要写“保持简洁”。
- 为每个关键规则寻找至少两个页面或组件证据。只有一个证据时降低置信度。

## 可借鉴的设计原型

以下原型用于帮助归类，不是套用厂商 token 的快捷方式：

| 原型 | 辨识度杠杆 | 常见风险 |
|---|---|---|
| 开发者极简 | 高对比中性色、几何无衬线、单色组件、技术型等宽标签 | 做成没有技术细节的普通黑白站 |
| 金融基础设施 | 冷色主调、轻字重、数据数字特性、渐变作为大尺度氛围 | 渐变滥用到每张卡片 |
| 人文编辑 | 暖色画布、衬线标题、人文无衬线正文、深色产品展示面 | 只换成米色而忽略字体节奏 |
| 高密度生产力 | 紧凑间距、低对比表面、精确状态、键盘和命令语义 | 营销式大标题破坏工作界面密度 |
| 创作工具 | 中性画布、面板与工具栏层级、彩色内容作为主角 | UI 装饰抢过用户内容 |
| 电影化展示 | 深色画布、全幅媒体、极少高亮色、克制控件 | 图片太暗或裁切导致主体不可辨识 |

`awesome-design-md` 的公开样例体现了这种分析深度：Vercel 样例把几何字体、等宽技术层、灰阶表面和大尺度多色渐变视为一套关系；Stripe 样例把深色文字、电光主色、轻字重、数字特性与营销/产品界面的明暗切换关联起来；Claude 样例则用暖画布、衬线展示字、珊瑚主操作和深色产品表面建立节奏。其 2026-04-08 之前的目录还包含 `preview.html` 与 `preview-dark.html`，用颜色、字体、按钮、卡片、表单、间距、圆角和 elevation 标本验证文档。借鉴其分析和验证方法，不要复制品牌结论或维护者署名。

## 预览反向验证

`preview.html` 与 `preview-dark.html` 必须直接消费或逐项映射 `DESIGN.md` 的规范值。预览不是“做一个像来源站点的首页”，而是回答以下问题：

- 颜色角色放在一起时是否仍有清晰层级和足够对比？
- 字体层级、行高和最长示例文字是否真实可用？
- 默认与已观察状态是否不改变控件尺寸、不造成布局跳动？
- 卡片、表单、导航和来源特有组件是否呈现同一套设计语法？
- 桌面与 390px 左右移动视口是否无横向溢出、遮挡和文本截断？

发现任一预览与文档冲突时，以证据优先级回到 `DESIGN.md` 修正，再同步两个预览；不要只在预览 CSS 中打补丁掩盖 token 问题。

## 反向验证

完成文档后，尝试仅依据该文档回答以下问题：

- 新页面的画布、容器、标题、正文和主按钮该使用哪些 token？
- 卡片靠什么与背景区分？边框、阴影还是表面色？
- 一个包含表单、错误提示和禁用按钮的流程如何保持同一语言？
- 移动端哪些布局会重排，哪些固定格式元素保持尺寸？
- 哪三个错误会让结果立刻失去来源风格？

无法明确回答时，回到证据补充规则，而不是堆更多形容词。
