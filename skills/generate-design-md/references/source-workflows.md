# 来源取证流程

## 先确定范围

先写一句范围声明，例如“只分析公开营销站，不包含登录后的产品界面”。如果用户没有限定，优先覆盖同一产品的核心表面，不要混合明显不同的年代、子品牌或活动页。

建立页面采样表，至少记录：来源、页面类型、视口、观察到的组件、可获取的技术证据、异常点。混合来源时用同一张表合并证据。

开始取证前创建一个系统临时目录，例如 `/tmp/generate-design-md-<任务标识>/`。浏览器截图、accessibility snapshot、computed-style JSON、下载的 HTML/CSS、证据表和调试日志都只能写入该目录；不要写入目标项目或最终交付目录。完成审计和视觉复核后删除该临时目录。

## 本地项目

1. 读取项目规则文件、包管理配置和前端入口，识别框架、CSS 方案、组件库与主题机制。
2. 优先检查全局 CSS、CSS variables、Tailwind 配置、theme/provider、字体加载、图标库和共享组件。
3. 搜索颜色、字号、字重、行高、间距、圆角、阴影、容器宽度和媒体查询的定义与调用位置。
4. 统计重复值及语义别名，确认哪些是有效 token，哪些只是局部例外。
5. 若项目可运行，启动现有开发命令并查看代表性路由。不得为了分析而重构项目。
6. 检查组件源码中的变体和状态，包括 focus-visible、disabled、loading、error、empty 和 responsive 分支。

本地源码中声明但从未使用的 token 只作为候选；真实组件中反复使用的值优先。

## 在线 URL

优先使用浏览器自动化或 Playwright 类工具查看真实页面。不要只读取搜索摘要。

1. 选择 4 至 8 个代表页面：首页、产品或功能页、定价页、文档页、登录或表单页，以及可访问时的产品界面。
2. 分别检查桌面和移动视口。推荐至少 1440x900、1024x768 和 390x844。
3. 记录 CSS variables、computed styles、加载字体、容器宽度、网格、媒体查询和交互状态。
4. 对按钮、链接、输入框、导航、卡片、表格、弹窗等触发 hover、focus、active 或展开状态。
5. 截取关键页面作为视觉证据。动态页面要等待字体、图片和布局稳定；截图保存到本次任务的临时取证目录。
6. 对受登录、地区、反爬或付费限制的表面明确记录不可见范围，不要绕过访问控制。

如果只能读取 HTML/CSS 而不能渲染页面，输出可以保留精确 token，但视觉气质与布局结论应降低置信度。

## 截图

1. 检查截图是否包含完整视口、裁切、缩放、设备像素比和深浅主题信息。
2. 从重复关系估算颜色角色、字号层级、间距基数、容器宽度、圆角和阴影，不要执着于单个像素。
3. 多张截图先对齐同类组件，再归纳稳定规则；一次性活动元素不要升级为系统 token。
4. 字体只能根据可见字形判断类别与近似替代，除非用户同时提供字体信息。
5. 单一视口无法证明断点。只写可观察布局，并把响应式策略标为建议或推断。
6. 看不见的组件状态不得写成“已观察事实”。可以提出保守的补全建议，但必须明确其来源是通用可用性原则。

## 混合来源

使用高置信来源校准低置信来源：源码确定 token，浏览器确认实际渲染，截图补充无法访问的状态。发生冲突时记录差异，优先采用当前实际渲染值，并说明源码可能存在旧 token 或主题覆盖。

## 取证记录

在分析过程中维护内部证据表：

| 规则 | 值或描述 | 来源位置 | 证据类型 | 置信度 | 备注 |
|---|---|---|---|---|---|
| 主按钮背景 | `#...` | 首页主 CTA | computed style | 高 | 三个页面一致 |

证据表始终是临时分析材料，不作为默认交付；将来源范围、置信度和关键限制浓缩进最终文档。只有用户明确要求原始证据时，才把指定证据作为额外交付，此时不再适用“只交付三个文件”的默认规则。
